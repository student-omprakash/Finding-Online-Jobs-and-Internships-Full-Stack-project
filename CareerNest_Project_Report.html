1. Introduction

In today's competitive digital economy, students and fresh graduates face a fragmented and overwhelming job search experience. Existing job portals are either too broad for entry-level seekers, lack integrated profile management, or provide no real-time visibility into the application process. Recruiters looking to hire freshers also struggle with the absence of a dedicated, structured platform focused on early-career talent.

CareerNest is a full-stack web application designed to bridge this gap. It serves as a centralised, role-based ecosystem where students can discover jobs and internships matching their skills, build comprehensive professional profiles, and track their application status in real time. Simultaneously, recruiters can post job listings, manage applicants, and update application statuses through a dedicated dashboard.

The application is built using the MERN stack (MongoDB, Express.js, React, Node.js), deployed live on Render cloud hosting, and backed by a complete DevOps infrastructure including Docker containerisation, Jenkins CI/CD pipeline, GitHub Actions workflows, and a Prometheus + Grafana monitoring stack.

1.1. Project Background

The project was conceived as a solution to a real and commonly faced challenge: the absence of a student-centric, all-in-one platform combining profile management, job discovery, real-time application tracking, and recruiter tools in a single system. CareerNest was developed incorporating not only the application layer but also a complete DevOps lifecycle — making it a comprehensive demonstration of production-grade software engineering.

1.2. Project Objectives

  • Build a scalable full-stack MERN web application with a clean REST API architecture.
  • Implement JWT-based authentication with role-based access control (Student, Recruiter, Admin).
  • Provide students a platform to discover jobs, build professional profiles, and track applications in real time.
  • Provide recruiters a dashboard to post job listings, manage applicants, and update application statuses.
  • Implement Docker Compose orchestration for MongoDB, Backend, Frontend, Prometheus, and Grafana.
  • Set up a Jenkins CI/CD pipeline with lint, build, syntax check, and Docker validation stages.
  • Integrate GitHub Actions for automated CI and CD (Render deployment health monitoring).
  • Implement Prometheus + Grafana real-time backend monitoring (HTTP metrics, CPU, memory).
  • Deploy the application live on Render cloud with MongoDB Atlas as the production database.

2. Scope of the Study / Problem Statement

2.1. Problem Statement

Students and freshers face several critical challenges with existing job and internship portals:

  • Fragmented Platforms: No single platform combines internships, full-time roles, profile management, and application tracking in a unified system.
  • Poor UX for Freshers: Most platforms are designed for experienced professionals and overwhelm entry-level seekers.
  • No Real-Time Application Tracking: Students apply and receive no feedback or visibility into their application status.
  • No Integrated Profile Builder: Students must maintain separate documents, portfolios, and links across different platforms.
  • Recruiter Gaps: No dedicated platform with structured applicant management for freshers hiring.
  • No Skill-Based Matching: Existing systems do not match job listings to student skill profiles.

2.2. Scope of Solution

CareerNest addresses all the above by implementing:

  • A unified role-based platform for Students and Recruiters with separate purpose-built dashboards.
  • A comprehensive student profile covering bio, education, experience, skills, certifications, resume upload, and social links.
  • Job listing with filters for type (Internship, Full-time, Part-time, Contract), location, skills, and experience level.
  • Real-time application tracking with five lifecycle statuses: Applied → Shortlisted → Interviewing → Accepted / Rejected.
  • Skill-based job recommendations from the student's profile skills.
  • A complete DevOps pipeline with Docker, Jenkins, GitHub Actions, Prometheus, and Grafana.

4. Software Requirement Analysis

4.1. General Description

CareerNest is a web application serving two user types — Students and Recruiters. The backend is a Node.js/Express REST API handling authentication, job management, profile management, and application tracking, backed by MongoDB. The frontend is a React SPA. The system is containerised with Docker and deployed on Render cloud hosting.

4.2. Functional Requirements

  • FR1 — Registration & Login: Users register as Student or Recruiter with email/password. A JWT token is returned on successful login.
  • FR2 — Password Recovery: Forgot Password flow using a time-limited OTP sent via Nodemailer (Gmail SMTP).
  • FR3 — Role-Based Access: Three roles — Student, Recruiter, Admin — each with separate API permissions.
  • FR4 — Job Listing & Search: Browse all open jobs filtered by type, location, skills, and experience level.
  • FR5 — Job Management: Recruiters create, edit, delete listings and toggle open/closed status.
  • FR6 — Student Profile: Bio, contact, education, experience, skills (with proficiency), certifications, social links, resume PDF upload.
  • FR7 — Job Application: One-click apply; unique compound index {job, applicant} prevents duplicates.
  • FR8 — Application Tracking: Students view all their applications with live status updates.
  • FR9 — Applicant Management: Recruiters view applicants per job and update statuses.
  • FR10 — Recommendations: Skill-matched job recommendations based on student profile.
  • FR11 — Health Check: /health endpoint returns uptime and environment status.
  • FR12 — Metrics: /metrics endpoint in Prometheus format for monitoring.

6. Module Description & Features

6.1. Authentication Module

The authentication system uses JSON Web Tokens (JWT) for stateless, scalable session management. Passwords are hashed using Bcrypt.js (salt: 10). A time-limited OTP-based password recovery flow is implemented via Nodemailer and Gmail SMTP.

  • Registration: Fields validated by express-validator, password bcrypt-hashed, JWT returned on success.
  • Login: Password validated via bcrypt.compare(); JWT token issued on success.
  • Protected Routes: Token verified by protect middleware; role checked by authorize() middleware.
  • OTP Reset: Time-limited OTP sent via Gmail SMTP → user verifies OTP → new password set and hashed.

6.2. Job & Internship Module

Provides complete CRUD for job listings with search and filtering by type, location, keyword, and experience level. Recruiters can create, edit, delete listings and toggle open/closed status. A recommendations endpoint returns skill-matched jobs from the student's profile.

6.3. Student Profile Module

Students can build a detailed professional profile stored in the Profiles collection (one-to-one with Users), covering:

  • Bio and contact details (phone, address, city, state, country, zip)
  • Social links: LinkedIn, GitHub, personal website, Twitter
  • Education history and work experience (multiple entries, with current toggle)
  • Skills with proficiency level (Beginner / Intermediate / Expert)
  • Certifications with issuing organisation, date, and URL
  • Resume PDF upload (handled by Multer, stored in /uploads)

6.4. Application Management Module

Manages the full lifecycle of a job application. A unique compound MongoDB index on {job, applicant} prevents duplicate applications at the database level. Status transitions: applied → shortlisted → interviewing → accepted / rejected. Students view all their applications and live statuses; recruiters view all applicants per job and update statuses with one click.

6.5. Recruiter Dashboard

Provides a consolidated view of all job listings with total applications received, per-job applicant lists, applicant profile and resume access, and one-click status updates. Recruiters can create, edit, delete job listings and toggle open/closed status from the dashboard.

7. DevOps, CI/CD & Monitoring

7.1. Docker Containerisation

The application is fully containerised. A docker-compose.yml file orchestrates five services:

ServiceImagePortPurpose
mongodbmongo:7.027017Local database with named volume; health check via mongosh ping
backendCustom Dockerfile (Node.js)5005Express.js API; starts only after MongoDB passes health check
frontendMulti-stage Dockerfile (Vite→Nginx)3000→80React SPA compiled by Vite, served by Nginx
prometheusprom/prometheus:v2.51.09090Scrapes /metrics every 5 seconds
grafanagrafana/grafana:10.4.14000→3000Monitoring dashboard connected to Prometheus

All services communicate via a named internal bridge network (careernest-network). Named volumes (mongodb_data, backend_uploads, grafana_data) ensure data persists across container restarts.

7.2. Jenkins CI/CD Pipeline

A declarative Jenkinsfile automates build and validation through seven sequential stages:

  1. Workspace Clean — Removes all artifacts from previous builds using cleanWs().
  2. Checkout Code — Fetches the latest codebase from Git using checkout scm.
  3. Install Dependencies — Installs npm packages for all workspace modules via npm run install:all.
  4. Frontend Lint — Runs ESLint on the React codebase to enforce code quality.
  5. Backend Syntax Check — Runs node --check src/index.js to validate JavaScript syntax without executing.
  6. Frontend Production Build — Compiles the React SPA with Vite into an optimised production bundle.
  7. Docker Compose Validation — Runs docker compose config to validate all five service configurations.

7.3. GitHub Actions Workflows

CI Workflow (ci.yml) — Triggers on push/PR to main or develop:

  • Job 1: ESLint frontend code quality check on Node.js 20 using npm ci
  • Job 2: Vite production build (runs only if Job 1 passes); verifies dist/ directory exists
  • Job 3: Backend Node.js syntax check (node --check src/index.js)

CD Workflow (cd.yml) — Triggers on push to main only:

  • Waits 90 seconds for Render's auto-deploy to pull and build the latest commit
  • Polls /health endpoint with 5 retries at 30-second intervals; handles cold starts gracefully
  • Outputs frontend and backend deployment URLs in the build summary

7.4. Prometheus & Grafana Monitoring

The backend exposes a /metrics endpoint via prom-client, scraped by Prometheus every 5 seconds (prometheus.yml). Grafana displays a pre-provisioned custom dashboard (careernest_dashboard.json) covering:

  • Default metrics: Heap memory, CPU usage, event loop lag, garbage collection statistics.
  • Custom metric — http_request_duration_seconds: Histogram tracking response times labelled by method, route pattern, and HTTP status code. Buckets: 0.1s, 0.3s, 0.5s, 0.9s, 1.5s, 3s, 5s. Route labels use req.route.path to avoid high cardinality from dynamic IDs.

10. Challenges & Solutions

#ChallengeSolution
1 CORS errors in production — frontend and backend on different Render domains blocked API calls. Implemented a dynamic CORS origin whitelist including the production Render URL and localhost. FRONTEND_URL env variable enables configuration without code changes.
2 Docker startup order — backend crashed before MongoDB was ready to accept connections. Added depends_on with condition: service_healthy and a MongoDB health check (mongosh ping) so backend only starts after database is ready.
3 Render free-tier cold starts — deployment health checks failed due to slow first response after inactivity. CD workflow uses a graceful retry loop (5 attempts × 30s) and treats timeout as a warning, as cold starts are expected on free-tier hosting.
4 Password field exposure — hashed passwords accidentally included in some API responses. Added select: false to the password field in the Mongoose User schema so it is never returned in any query unless explicitly requested.
5 Duplicate job applications — no prevention mechanism for a student applying to the same job twice. Added a unique compound MongoDB index on {job: 1, applicant: 1} enforcing uniqueness at the database level with a clear API error message.
6 High-cardinality Prometheus metrics — dynamic URL paths (e.g., /api/jobs/64abc...) as labels created excessive time series. Metrics middleware uses req.route.path (pattern /api/jobs/:id) instead of the actual URL, keeping the number of unique label combinations manageable.
7 Ephemeral file storage on Render — uploaded resume files lost on each new deployment. Named Docker volume (backend_uploads) persists files locally. In production, the file path is stored as a relative string in MongoDB, allowing easy migration to cloud storage (e.g., AWS S3).

11. Future Enhancements & Conclusion

11.1. Future Enhancements

  • AI-Based Recommendations: Integrate a skill-matching algorithm or ML model for more accurate job recommendations and resume scoring.
  • Real-Time Notifications: WebSocket-based (Socket.io) notifications so students are instantly alerted on application status changes.
  • Automated Testing Suite: Unit and integration tests using Jest and Supertest for all API routes.
  • Redis Caching: Cache frequently accessed job listings to reduce database load under high traffic.
  • Rate Limiting: express-rate-limit to protect the API against brute-force and excessive requests.
  • Kubernetes Deployment: Migrate from Docker Compose to Kubernetes with HPA for production-grade horizontal scaling.
  • Cloud File Storage: Replace local disk resume storage with AWS S3 or Cloudinary for persistent, scalable delivery.
  • Analytics Dashboard: Student analytics (applications sent, shortlist rate); recruiter analytics (time-to-hire, applicant funnel).

11.2. Conclusion

CareerNest successfully demonstrates the design and implementation of a production-grade, full-stack MERN web application addressing a real-world problem in student career discovery. The platform bridges the gap between freshers and recruiters by providing a unified, intuitive ecosystem for job discovery, profile management, and application tracking.

The project covers every layer of modern software development — database schema design, REST API architecture, responsive React UI, file management, OTP-based email authentication, security hardening, and cloud deployment. The addition of a complete DevOps stack comprising Docker multi-service orchestration, a Jenkins CI/CD pipeline, GitHub Actions automated workflows, and a Prometheus + Grafana real-time monitoring dashboard elevates CareerNest beyond a typical academic project into a showcase of industry-standard engineering practices.

The application is deployed and accessible as a live service on Render, demonstrating the ability to take a concept through design, development, testing, containerisation, and production deployment in a systematic and professional manner. Through this project, comprehensive practical knowledge was acquired in full-stack JavaScript development, NoSQL data modelling, RESTful API design, Docker containerisation, CI/CD automation, and real-time system monitoring.
